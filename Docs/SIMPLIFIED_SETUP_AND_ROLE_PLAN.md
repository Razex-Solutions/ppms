# PPMS Simplified Setup And Role Plan

## Purpose
This document defines the new preferred direction for:
- brand setup
- organization setup
- station setup
- tank/dispenser/nozzle setup
- invoice identity setup
- role creation logic
- single-station vs multi-station hierarchy
- question-based Flutter onboarding/setup flow

This plan should be treated as the next major setup/configuration redesign layer on top of the current PPMS foundation.

---

## 1. Core Direction

The setup experience should become:
- simpler
- question-based
- guided
- inheritance-first
- less form-heavy
- easier for real customers to understand

Instead of showing large raw forms first, the app should ask structured questions and build the right records underneath.

Examples:
- choose brand
- enter organization details
- choose number of stations
- if 1 station, simplify roles and inheritance
- if multiple stations, enable broader hierarchy
- choose number of tanks
- choose number of dispensers
- choose nozzle counts and mappings

The UI should feel like a guided setup wizard, not a raw admin panel.

## 10. Role Logic Redesign

The current hierarchy should be simplified based on organization shape.

### Top platform role
- `MasterAdmin`
  - Razex Solutions platform owner
  - can create organizations
  - can manage subscriptions
  - can support/edit any tenant when needed

### Below MasterAdmin
Role creation should depend on single-station vs multi-station setup.

---

## 11. Single-Station Role Logic

If organization has only 1 station:
- do not create unnecessary layered admin roles
- avoid separate organization admin and station admin unless truly needed

### Preferred simplified structure
- `MasterAdmin`
- `Admin`
- `Manager`
- `Accountant`
- `Operator`
- `Profile-only staff`

### Why
If there is only one station:
- organization admin
- station admin
- duplicate admin layers

become mostly redundant.

In that case, one tenant admin role is enough beneath MasterAdmin.

### Single-station rule
If `station_count == 1`
- create a single tenant admin role path
- hide unnecessary higher tenant layers
- simplify dashboards and permission setup

---

## 12. Multi-Station Role Logic

If organization has more than 1 station:
- broader hierarchy becomes useful

### Preferred structure
- `MasterAdmin`
- `HeadOffice` or `OrgAdmin`
- `StationAdmin`
- `Manager`
- `Accountant`
- `Operator`
- `Profile-only staff`

### Why
Multi-station businesses benefit from:
- organization-level oversight
- station-level administration
- cleaner delegation

### Multi-station rule
If `station_count > 1`
- enable org-level admin role
- enable station admin role
- allow scoped station assignment

---

## 13. Role Creation Rules

### MasterAdmin
Can:
- create organization
- assign subscription/trial
- configure platform support access
- create top tenant admin

### If single station
MasterAdmin creates:
- `Admin`

Then `Admin` creates:
- `Manager`
- `Accountant`
- `Operator`
- profile-only staff

### If multiple stations
MasterAdmin creates:
- `HeadOffice` or `OrgAdmin`

Then `HeadOffice` / `OrgAdmin` creates:
- `StationAdmin`
- optionally `Manager`, `Accountant`, `Operator`

Then `StationAdmin` creates:
- station staff
- manager
- operator
- accountant
- profile-only staff

## 14. Flutter Setup Flow Direction

The Flutter setup experience should become question-based.

### Instead of this
- many raw forms shown at once

### Do this
- ask one structured question block at a time
- reveal next step based on answer
- auto-generate records where sensible
- provide summary/review before saving

### Example flow
1. Choose brand
2. Enter organization details
3. Decide legal name behavior
4. Choose station count
5. If 1 station:
   - inherit organization details?
6. If multiple:
   - configure each station
7. Choose tank count
8. Configure tanks
9. Choose dispenser count
10. Configure dispensers
11. Choose nozzle counts
12. Configure nozzle mappings
13. Configure invoice identity
14. Review and finish

This is much better than large forms.

---

## 15. Why This Approach Is Better

This approach is better because it is:
- cleaner
- minimal
- easier for users
- more real-world accurate
- easier to validate
- easier to scale
- better for franchise/multi-station setups
- better for single-station simplicity

It also reduces:
- form fatigue
- duplicate entry
- confusing role hierarchy
- unnecessary setup complexity
